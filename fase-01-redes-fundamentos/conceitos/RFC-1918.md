# RFC 1918 — Faixas de IP Privado

RFC 1918 e o documento que reservou 3 faixas de IP para uso privado. IPs dessas faixas nao roteiam na internet publica — existem so dentro de redes locais.

## Por que existe

IPv4 tem ~4 bilhoes de enderecos. Insuficiente para um IP unico por dispositivo no mundo todo. Solucao: redes locais reutilizam IPs privados sem conflito, porque cada rede e isolada.

**Analogia — ramal interno da empresa:**
- Ramal 100 da sua empresa nao conflita com o ramal 100 da empresa vizinha — sao internos.
- Para ligar de fora, voce disca o numero publico da empresa; a recepcao repassa pro ramal.
- IP privado = ramal interno. IP publico = numero da empresa. NAT = recepcao que traduz.

Ao sair da rede local, o NAT do roteador **traduz** o IP privado pelo IP publico que a operadora atribuiu ao roteador. Ao voltar, faz o caminho inverso.

## As 3 faixas

| Classe | Primeiro octeto | Faixa privada | Regra de identificacao | Exemplo |
|--------|----------------|---------------|----------------------|---------|
| A | 1 — 126 | 10.0.0.0/8 | Primeiro octeto = **10** | `10.50.200.5` (AWS VPC) |
| B | 128 — 191 | 172.16.0.0/12 | Primeiro octeto = 172, segundo octeto **16 a 31** | `172.17.0.1` (Docker) |
| C | 192 — 223 | 192.168.0.0/16 | Primeiros dois octetos = **192.168** | `192.168.0.19` (rede domestica) |

> **127 nao e Classe B.** E reservado para loopback (`127.0.0.1` = localhost, seu proprio computador). Classe B comeca em **128**.

## Armadilha da faixa B

Nao e todo `172.x.x.x` que e privado — so o segundo octeto **entre 16 e 31**:

```
172.16.x.x → privado ✅
172.20.x.x → privado ✅
172.31.x.x → privado ✅ (limite da faixa)
172.15.x.x → publico ❌ (abaixo de 16)
172.35.x.x → publico ❌ (acima de 31)
```

## Subnets dentro das faixas privadas

O CIDR da faixa privada define um **espaco**. Qualquer subnet dentro desse espaco e privada, independente da mascara usada:

- `10.10.0.0/16` → privado ✅ (primeiro octeto = 10, dentro do 10.0.0.0/8)
- `172.17.0.0/16` → privado ✅ (segundo octeto 17, dentro do 172.16.0.0/12)
- `192.168.1.0/24` → privado ✅ (dentro do 192.168.0.0/16)

**Regra:** se o IP esta dentro do espaco da faixa RFC 1918, e privado. A mascara da subnet nao muda isso.

## Por que DevOps importa

- **AWS:** VPCs usam `10.0.0.0/16` (dentro da faixa A privada).
- **Docker:** redes internas em `172.17.0.0/16` (faixa B privada).
- **Qualquer rede domestica:** `192.168.x.x` (faixa C privada).

Voce vai ver essas faixas toda vez que configurar rede em container, VM ou cloud.

## 🇺🇸 English

RFC 1918 is the document that reserved 3 ranges of IP addresses for private use. IPs in these ranges do not route on the public internet — they exist only inside local networks.

## Why it exists

IPv4 has ~4 billion addresses. Not enough for a unique IP per device worldwide. Solution: local networks reuse private IPs without conflict, because each network is isolated.

**Analogy — a company's internal extension:**
- Extension 100 at your company does not conflict with extension 100 at the company next door — they are internal.
- To call from outside, you dial the company's public number; reception forwards to the extension.
- Private IP = internal extension. Public IP = the company number. NAT = the reception that translates.

When leaving the local network, the router's NAT **translates** the private IP into the public IP the ISP assigned to the router. On the way back, it does the reverse.

## The 3 ranges

| Class | First octet | Private range | Identification rule | Example |
|--------|----------------|---------------|----------------------|---------|
| A | 1 — 126 | 10.0.0.0/8 | First octet = **10** | `10.50.200.5` (AWS VPC) |
| B | 128 — 191 | 172.16.0.0/12 | First octet = 172, second octet **16 to 31** | `172.17.0.1` (Docker) |
| C | 192 — 223 | 192.168.0.0/16 | First two octets = **192.168** | `192.168.0.19` (home network) |

> **127 is not Class B.** It is reserved for loopback (`127.0.0.1` = localhost, your own computer). Class B starts at **128**.

## The Class B trap

Not every `172.x.x.x` is private — only when the second octet is **between 16 and 31**:

```
172.16.x.x → private ✅
172.20.x.x → private ✅
172.31.x.x → private ✅ (range limit)
172.15.x.x → public ❌ (below 16)
172.35.x.x → public ❌ (above 31)
```

## Subnets inside the private ranges

The private range's CIDR defines a **space**. Any subnet inside that space is private, regardless of the mask used:

- `10.10.0.0/16` → private ✅ (first octet = 10, inside 10.0.0.0/8)
- `172.17.0.0/16` → private ✅ (second octet 17, inside 172.16.0.0/12)
- `192.168.1.0/24` → private ✅ (inside 192.168.0.0/16)

**Rule:** if the IP is inside the RFC 1918 range space, it is private. The subnet mask does not change that.

## Why it matters for DevOps

- **AWS:** VPCs use `10.0.0.0/16` (inside the private Class A range).
- **Docker:** internal networks in `172.17.0.0/16` (private Class B range).
- **Any home network:** `192.168.x.x` (private Class C range).

You will see these ranges every time you configure networking in a container, VM or cloud.
