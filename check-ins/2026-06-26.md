## 🇬🇧 English

**Subnet drill**
- `/26` → 6 host bits (`2^6=64` addresses), 62 usable, `256-64=192` → `255.255.255.192`, `26-24=2` → 4 subnets. ✅ Nailed first try, anchoring on host bits first (the exact spot that slipped on the `/27` the day before).

**Concept — MTU > packet (the two outcomes)**
- **Without DF:** the router fragments, the destination reassembles.
- **With DF:** the router **drops** the packet and returns an ICMP "Fragmentation Needed" with the maximum size. (Path MTU Discovery named, deferred.)

**Recon (terminal)**
- `ip link show wlp0s20f3` → MTU 1500, state UP.
- `ip route | grep default` → gateway `192.168.0.1`, IP `192.168.0.24`, `proto dhcp`.

**Lab 2.4 Stage 2 — live fragmentation with tcpdump**
- Setup: window 1 `sudo tcpdump -i wlp0s20f3 -n -v host 192.168.0.1`; window 2 `ping -c 3 -s 2000 192.168.0.1` (payload 2000 > MTU 1500, **no** `-M do` → allows fragmentation).
- Textbook capture: each echo request left as **2 fragments with the same `id`** — piece 1 `offset 0, flags [+], length 1500`; piece 2 `offset 1480, flags [none], length 548`. The reply came back fragmented too → both sides fragment.
- Read live: same `id` = same parcel (reassembly key); `flags [+]` = More Fragments; `flags [none]` = last piece; `offset` = position in the whole. The ICMP header rides only in the 1st fragment (the 2nd shows as `ip-proto-1`, anonymous filler). Bonus: caught an IGMP with `flags [DF]` on screen — visual contrast `[+]` (splits) vs `[DF]` (won't split).
- Applied check: if fragment 2 were lost, the destination can't reassemble and discards the whole packet — total loss, not "half".
